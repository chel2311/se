📖 デバッグの基本

デバッグは、プログラムが期待通りに動作しない原因を特定し、修正する重要な作業です。 IDEのデバッガ プログラムの実行を制御し、内部状態を調査するためのツール。ブレークポイントの設定、変数の監視、ステップ実行などの機能を提供します。 を使うことで、効率的にバグを発見できます。

よくあるバグの種類：

  • 構文エラー - 文法の誤り（；）
  • 論理エラー - ロジックの誤り（条件式の間違い）
  • 実行時エラー - 実行中に発生（範囲外エラー）
  • 型エラー - データ型の不一致

ステップ実行（ステップイン・ステップオーバー）

Step Execution (Step Into / Step Over)
デバッグ中にプログラムを1行ずつ実行する機能。 ステップイン メソッドや関数の呼び出しがある行で使用すると、そのメソッドの中に入って最初の行から実行を続けます。内部の動作を詳しく調べたい時に使います。 はメソッドの中に入り、 ステップオーバー 現在の行を実行して次の行に進みます。メソッド呼び出しがあっても、その中には入らずに結果だけを得て次に進みます。 はメソッドを実行して次の行へ進みます。

📖 ステップ実行の基本概念

ステップ実行は、プログラムの実行を細かく制御する機能です。 ブレークポイントで停止した後、1行ずつプログラムを進めながら、変数の変化や処理の流れを詳しく観察できます。

function main() { const x = 10; const y = 20; const result = add(x, y); // ← ここで選択が必要 console.log(result); } function add(a, b) { const sum = a + b; return sum; }

⬇️ ステップイン（Step Into）

メソッドや関数の呼び出しがある行で使用すると、そのメソッドの中に入って最初の行から実行を続けます。 メソッドの内部動作を詳しく調べたい時に使用します。

使用場面：
  • 自分が書いたメソッドの動作を確認したい時
  • 引数がどのように処理されるか追跡したい時
  • 複雑な処理の流れを理解したい時

➡️ ステップオーバー（Step Over）

現在の行を実行して次の行に進みます。メソッド呼び出しがあっても、その中には入らずに結果だけを得て次に進みます。

使用場面：
  • 信頼できるライブラリメソッド 標準ライブラリや外部ライブラリが提供するメソッド。内部の実装は複雑で、通常はデバッグの対象外です。 の内部は見る必要がない時
  • プログラムの大まかな流れを追いたい時
  • 特定の部分だけに注目したい時

イミディエイトウィンドウ

Immediate Window
デバッグ中に 式を評価 プログラムの式や計算を実行して結果を確認すること。例：2 + 3 * 4 の結果を計算したり、変数の値を確認したりできます。 したり、変数の値を確認・変更できる対話型ウィンドウ。「即座に」結果を確認できることから、この名前が付いています。

📖 イミディエイトウィンドウとは

イミディエイトウィンドウは、デバッグ中にリアルタイムでコードを実行できる強力なツールです。 変数の現在値を確認したり、メソッドを呼び出したり、さらには変数の値を変更してプログラムの動作をテストすることもできます。

イミディエイトウィンドウでできること：

// 変数の値を確認 ? userName "田中太郎" // 式を評価 ? age >= 20 && hasLicense true // メソッドを呼び出し ? calculateTax(1000) 108 // 変数の値を変更 userName = "佐藤花子" "佐藤花子"

🎯 イミディエイトウィンドウの活用方法

  • 複雑な条件式のテスト

    if文の条件が複雑な場合、その条件式を貼り付けて結果を確認

  • メソッドの動作確認

    異なる引数でメソッドを呼び出し、結果を比較

  • デバッグ中の値の修正

    バグの原因となっている値を一時的に変更して、修正後の動作を確認

Build & Compile
コンパイル 人間が読み書きできるソースコード（高級言語）を、コンピュータが理解できる機械語（低級言語）に変換する処理。 はソースコードを機械語に変換する処理。 ビルド コンパイルを含む、実行可能なプログラムを作成する全体のプロセス。リンク、リソースの埋め込み、最適化なども含まれます。 はコンパイルを含む、実行可能なプログラムを作成する全体のプロセスです。

📖 コンパイルとは

コンパイルは、人間が読み書きできるソースコード（高級言語）を、 コンピュータが理解できる 機械語 CPUが直接実行できる命令の集まり。0と1の羅列（バイナリ）で表現され、人間には理解困難ですが、コンピュータには最も効率的です。 （低級言語）に変換する処理です。

// ソースコード（人間が理解できる） public class HelloWorld { public static void Main() { Console.WriteLine("Hello, World!"); } } ↓ コンパイル // 機械語（コンピュータが理解できる） 01001000 01100101 01101100 01101100 01101111 ...

🔨 ビルドプロセスの全体像

ビルドは、コンパイルを含むより大きなプロセスです。 ソースコードから最終的な実行可能ファイルを作成するまでのすべての工程を含みます。

ビルドプロセスの主な工程：
  1. 構文チェック - コードの文法エラーを検出
  2. コンパイル - ソースコードを中間コードに変換
  3. リンク - 必要なライブラリ 再利用可能なコードの集まり。他の開発者が作成した機能を利用できます。例：数学計算ライブラリ、データベース接続ライブラリなど。 と結合
  4. リソース埋め込み - 画像や設定ファイルを組み込み
  5. 最適化 - 実行速度やファイルサイズの最適化
  6. 出力 - 実行可能ファイル（.exe、.dll）の生成

リファクタリング

Refactoring
プログラムの外部から見た動作は変えずに、内部のコード構造を改善する作業。読みやすく、保守しやすいコードにすることが目的です。

📖 リファクタリングとは

リファクタリングは、コードの「動作」は変えずに「品質」を向上させる作業です。 技術的負債 急いで書いた汚いコードや、設計の悪さなど、後で修正が必要になる問題。放置すると開発速度が低下し、バグが増える原因になります。 を減らし、将来の開発を効率化します。

リファクタリングの例：

😰 改善前

// 読みにくく、意図が不明確 if (u.a >= 20 && u.a < 65 && u.j != null && u.i > 300000) { // 処理 }

😊 改善後

// 意図が明確で読みやすい const isWorkingAge = user.age >= 20 && user.age < 65; const hasJob = user.job != null; const hasEnoughIncome = user.income > 300000; if (isWorkingAge && hasJob && hasEnoughIncome) { // ローン申請可能 }

🎯 主なリファクタリング手法

  • メソッド抽出

    長いメソッドを小さな意味のある単位に分割

  • 変数名の改善

    意味不明な変数名を分かりやすい名前に変更

  • 重複コードの削除

    同じ処理を1か所にまとめる

  • 条件式の簡略化

    複雑な条件を分かりやすく整理

リファクタリングを実践してみよう

実際のコードをリファクタリングして、どのように改善されるか確認しましょう。

課題：以下のコードをリファクタリングしてください
// 価格計算（読みにくいコード） function calc(p, q, t) { let r = p * q; if (q > 10) { r = r * 0.9; // 10個以上で10%割引 } if (t == "member") { r = r * 0.95; // 会員は5%割引 } return r; }
⚠️
リファクタリングの注意点：
  • 必ずテスト プログラムが正しく動作することを確認する作業。リファクタリング前後で動作が変わっていないことを保証するために重要です。 を用意してから行う
  • 一度に大きく変更しない（小さなステップで）
  • 動作が変わっていないことを確認する

デプロイ

Deploy
開発したアプリケーションを 本番環境 実際にユーザーが使用する環境。開発環境とは異なり、高い信頼性とセキュリティが求められます。サーバー、データベース、ネットワークなどが含まれます。 に配置し、実際にユーザーが使える状態にすること。配備、展開とも呼ばれます。

📖 デプロイの基本概念

デプロイは、開発環境で作成したアプリケーションを、実際の運用環境（本番環境）に移す作業です。 単にファイルをコピーするだけでなく、環境設定、データベースの準備、セキュリティ設定なども含まれます。

デプロイの流れ：

  1. ビルド - リリース用の最適化されたビルドを作成
  2. テスト - ステージング環境 本番環境と同じ構成のテスト環境。本番デプロイ前の最終確認に使用します。 でのテスト
  3. 準備 - 必要なファイルやリソースの準備
  4. 転送 - サーバーへのファイル転送
  5. 設定 - 環境変数やデータベース接続の設定
  6. 確認 - 正常に動作することを確認

方法 特徴 適用場面

🚀 現代的なデプロイプロセス

CI/CD（継続的インテグレーション/継続的デリバリー）：

コードの変更を自動的にテスト・ビルド・デプロイする仕組み。開発効率と品質を大幅に向上させます。

デプロイプロセスをシミュレーション

モダンな自動デプロイの流れを体験してみましょう。

💡
デプロイのベストプラクティス：
  • 本番環境と同じ構成のステージング環境でテスト
  • データベースのバックアップを必ず取る
  • ロールバック デプロイに失敗した場合に、以前のバージョンに戻す作業。必ず準備しておくべき重要な手順です。 （元に戻す）手順を準備
  • デプロイ時刻はユーザーが少ない時間帯を選ぶ
